# MTProto Proxy Installer (`ee` + Real TLS)

> 🚀 Универсальный bash-скрипт для развёртывания защищённого MTProto-прокси на базе `9seconds/mtg` с поддержкой `ee`-секрета (Fake TLS) и реальными TLS-сертификатами от Let's Encrypt.

---

## 📋 О проекте

Этот скрипт автоматизирует полную настройку self-hosted прокси для доступа к Telegram из регионов с ограничениями. Поддерживает два режима работы:

| Режим | Описание |
|-------|----------|
| **🌐 Домен** | Укажите ваш домен (например, `proxy.yoursite.com`) — скрипт автоматически запросит и установит реальный сертификат |
| **🔢 Публичный IP** | Укажите статический публичный IP — скрипт запросит сертификат непосредственно на адрес (поддерживается Let's Encrypt с 2024 г.) |

✅ **Ключевые особенности:**
- Нативная поддержка `ee`-секретов (Fake TLS + SNI-проверка) для обхода DPI
- Реальные TLS-сертификаты (не самоподписные) — снижают вероятность детекта ТСПУ
- Автообновление сертификатов через cron + автоматический перезапуск прокси
- Управление через параметры: `--show-key`, `--renew-key`
- Минимальные требования: Debian 12/13, root-доступ, открытые порты 80/443
- Полная совместимость с **официальным клиентом Telegram** (без модификаций)

---

## ⚙️ Требования

- **ОС**: Debian 12 или 13 (другие дистрибутивы не тестировались)
- **Права**: запуск от `root` или через `sudo`
- **Сеть**:
  - Публичный статический IP-адрес
  - Открытые порты: `80/tcp` (для получения сертификата), `443/tcp` (для работы прокси)
- **Домен или IP**:
  - Для доменного режима: настроенная `A`-запись, указывающая на сервер
  - Для IP-режима: публичный адрес, не входящий в приватные диапазоны (`10.0.0.0/8`, `192.168.0.0/16` и т.д.)

## 🚀 Быстрый старт

### 1. Скачайте и подготовьте скрипт
```bash
# На сервере (под root)
curl -O https://your-repo/install-mtg-ee.sh
chmod +x install-mtg-ee.sh
```

### 2. Запустите установку
```bash
./install-mtg-ee.sh
```

Скрипт интерактивно запросит:
- Домен **или** публичный IP-адрес
- Автоматически выполнит:
  - Установку Docker и зависимостей
  - Запрос реального TLS-сертификата (Let's Encrypt)
  - Генерацию `ee`-секрета в HEX-формате
  - Создание конфигурации и запуск прокси
  - Настройку автообновления сертификатов

### 3. Настройте Telegram
После завершения скрипт выведет готовую строку подключения:

```
🔑 Ваш ee-секрет для Telegram:
┌─────────────────────────────────────────┐
│ ee473ce5d4958eb5f968c87680a23854a0676f6f676c652e636f6d
└─────────────────────────────────────────┘

📱 Настройки для официального Telegram:
  • Тип прокси: MTProto
  • Сервер:     <ваш-домен-или-IP>
  • Порт:       443
  • Секрет:     ee473ce5d4958eb5f968c87680a23854a0676f6f676c652e636f6d
```

В официальном клиенте:
```
Настройки → Данные и память → Прокси → Добавить прокси → MTProto
```

## 🎛️ Управление после установки

### Показать текущий ключ
```bash
./install-mtg-ee.sh --show-key
```
Выведет текущий `ee`-секрет и параметры для подключения без повторной генерации.

### Ротация ключа (рекомендуется каждые 30–45 дней)
```bash
./install-mtg-ee.sh --renew-key
```
- Сгенерирует новый `ee`-секрет
- Обновит конфигурацию
- Перезапустит прокси
- Выведет новый ключ для обновления в клиентах

> ⚠️ После ротации **все пользователи должны обновить настройки прокси** в Telegram — старый секрет перестанет работать.

### Дополнительные команды
```bash
# Просмотр логов в реальном времени
docker logs -f mtg-proxy

# Перезапуск прокси
cd /opt/mtg-ee-proxy && docker compose restart

# Проверка статуса
docker compose ps
```

## 🔐 Почему реальный сертификат важнее самоподписного?

| Критерий | Самоподписной | Реальный (Let's Encrypt) |
|----------|---------------|--------------------------|
| **Issuer в сертификате** | `Self-Signed` — явный индикатор прокси | `Let's Encrypt` — как у 90% легитимных сайтов |
| **Цепочка доверия** | Отсутствует — сразу флаг для анализаторов | Полная, валидная в ОС/браузерах |
| **Поведение при сканировании** | Может возвращать ошибки рукопожатия | Имитирует обычный HTTPS-сервер |
| **Репутация хоста** | Подозрительно для ТСПУ | Нейтрально, как веб-хостинг |

> 🎯 **Важно**: Реальный сертификат **не гарантирует** 100% невидимость, но **значительно снижает** вероятность автоматической классификации на этапе пассивного анализа трафика.

## ⚠️ Ограничения для сертификатов на IP-адреса

1. **Только публичные статические IP**  
   Let's Encrypt не выдаёт сертификаты на:
   - Приватные адреса (`10.0.0.0/8`, `192.168.0.0/16`, `172.16.0.0/12`)
   - Динамические / DHCP-адреса
   - Адреса из чёрных списков (Spamhaus и аналоги)

2. **Требуется доступность порта 80**  
   Для челленджа `http-01` порт должен быть открыт и доступен из интернета:
   ```bash
   # Проверка
   ss -tlnp | grep :80
   # Если используется ufw:
   ufw allow 80/tcp
   ```

3. **Лимиты Let's Encrypt**
   - 5 сертификатов на один домен/IP в неделю
   - 50 сертификатов на аккаунт в неделю
   - Для одного сервера обычно не является ограничением

4. **Если сертификат не выдаётся**:
   - Проверьте, не блокирует ли хостинг порт 80
   - Убедитесь, что `TARGET` резолвится в текущий публичный IP
   - Повторите запуск через 10–15 минут (возможны временные сбои ACME)

## 🛠️ Диагностика и отладка

### Базовая проверка
```bash
# Контейнер запущен?
docker ps | grep mtg-proxy

# Порт 443 слушается?
ss -tlnp | grep :443

# Логи прокси
docker logs mtg-proxy --tail=50
```

### Проверка TLS-сертификата
```bash
# Для домена
openssl s_client -connect your-domain.com:443 -servername your-domain.com </dev/null 2>/dev/null | openssl x509 -noout -issuer -subject

# Для IP (без SNI)
openssl s_client -connect YOUR_IP:443 </dev/null 2>/dev/null | openssl x509 -noout -issuer -subject
```
Ожидаемый вывод:
```
issuer=CN = R3, O = Let's Encrypt, C = US
subject=CN = your-domain.com   # или IP
```

### Проверка скрытности (с внешнего хоста)
```bash
# Эмуляция сканирования: должен быть валидный TLS-хендшейк
curl -v --resolve your-domain.com:443:YOUR_IP https://your-domain.com

# Ожидайте:
# * SSL connection using TLSv1.3 / AEAD-CHACHA20-POLY1305-SHA256
# * Server certificate: subject=... issuer=Let's Encrypt...
# *  SSL certificate verify ok.
```

> ❌ Если видите `MTProto`, `connection reset`, `plaintext` — прокси детектируется. Проверьте:
> - Используется ли `ee`-секрет (не `dd`, не base64)
> - Порт `443` и `network_mode: host` в compose
> - Сертификат привязан к правильному домену/IP

## 🔄 Обновление и миграция

### Обновление образа прокси
```bash
cd /opt/mtg-ee-proxy
docker compose pull
docker compose up -d
```

### Миграция на новый сервер
1. На старом сервере создайте бэкап:
   ```bash
   tar -czf /root/mtg-backup.tar.gz /opt/mtg-ee-proxy /etc/letsencrypt
   ```
2. Перенесите архив на новый сервер
3. Распакуйте и убедитесь, что домен/IP указывает на новый сервер
4. Запустите:
   ```bash
   cd /opt/mtg-ee-proxy && docker compose up -d
   ```

> 💡 Секрет и сертификаты сохраняются — клиенты Telegram не потребуют перенастройки.

## ❓ Частые вопросы

### ❓ Почему используется `network_mode: host`?
Для корректной обработки SNI в Fake TLS-режиме (`ee`) контейнер должен получать «сырой» TLS-хендшейк. Нативная трансляция портов Docker может искажать заголовки, что приводит к сбоям при подключении.

### ❓ Можно ли использовать другой порт вместо 443?
Технически — да, но **не рекомендуется**. Системы DPI в РФ активно сканируют нестандартные порты на предмет сигнатур MTProto. Порт `443` + `ee`-секрет — единственная комбинация, прошедшая массовое тестирование в 2024–2026 гг.

### ❓ Что делать, если Let's Encrypt исчерпал лимиты?
- Подождите 7 дней (сброс недельного лимита)
- Используйте альтернативный провайдер: ZeroSSL (требует регистрации)
- Временно используйте режим `dd` (базовый Fake TLS), но учтите: он менее устойчив к детекту

### ❓ Прокси работает, но соединения нестабильны
1. Проверьте нагрузку на сервер: `htop`, `nload`
2. Убедитесь, что провайдер не ограничивает исходящий трафик
3. Добавьте в `config.toml`:
   ```toml
   rate-limit = 0          # Снять лимит скорости
   buffer-size = 131072    # Увеличить буфер (опционально)
   ```
